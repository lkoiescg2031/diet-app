# AI Governance

이 문서는 프로젝트 AI 하네스의 변경 규칙을 정의한다.

## 기본 원칙

- 하네스는 프로젝트를 돕기 위한 장치이며, 그 자체가 과도한 일이 되면 안 된다.
- agent와 skill은 저장소 산출물로 관리한다.
- 모든 agent/skill 변경은 registry와 함께 변경한다.
- 장기 영향이 있는 결정은 ADR로 남긴다.
- 사용자 요청과 승인된 spec이 항상 우선한다.

## 변경 흐름

1. 변경 요청을 한 문장으로 정리한다.
2. `agents/ai-steward-agent.md`를 기준으로 새 artifact가 필요한지 판단한다.
3. 기존 agent/skill 수정으로 충분하면 새 파일을 만들지 않는다.
4. 새 artifact가 필요하면 목적, 사용 조건, 입력, 출력, 금지 사항을 작성한다.
5. 관련 registry를 같은 변경에서 갱신한다.
6. 장기 영향이 있으면 `docs/decisions/`에 ADR을 추가한다.
7. `docs/context-budget.md`에 읽기 비용 변화가 있는지 확인한다.

## 새 agent 승인 기준

- 기존 agent와 명확히 다른 책임이 있다.
- 반복적으로 호출될 가능성이 있다.
- 입력과 출력이 분명하다.
- 소유하거나 자주 다룰 파일이 설명된다.
- registry 항목을 만들 수 있다.

## 새 skill 승인 기준

- 프로젝트 특화 절차다.
- 같은 절차가 반복될 가능성이 있다.
- 절차를 빠뜨리면 품질, 데이터 안전성, 검증 신뢰도가 떨어진다.
- `SKILL.md`가 500줄 이하로 유지될 수 있다.
- 전역 skill과 중복되면 프로젝트 특화 차이를 설명할 수 있다.

## 병합 또는 폐기 기준

- 2개 이상의 agent가 같은 판단을 반복하면 병합 후보로 본다.
- 90일 이상 쓰이지 않고 대체 경로가 있으면 폐기 후보로 본다.
- 폐기 시 registry에서 제거하거나 폐기 상태를 표시하고 migration note를 남긴다.

## ADR 필요 기준

다음 변경은 ADR이 필요하다.

- 프론트엔드 스택 선택
- 로컬 저장 기술 선택
- 영속 데이터 schema versioning 변경
- migration 전략 변경
- 테스트 전략 변경
- agent/skill 모델의 큰 구조 변경

## 금지 사항

- registry 없는 agent/skill 추가
- 단발성 작업을 skill로 승격
- 임시 브레인스토밍 산출물 커밋
- 검증 없이 완료 상태 주장
